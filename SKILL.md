# setup-tool

这是跨宿主的工具接入顾问。按以下五个阶段工作，任何阶段信息不足时明确说明缺口，而不是猜测。

## 0. 确定宿主与边界

确认用户希望工具服务于 Claude Code、Codex、Hermes、OpenClaw、普通终端/项目，还是多个宿主。仅当当前上下文不能确定时询问。

读取 `references/install-routes.md`。只有官方资料已核验的平台 adapter 才能提供该宿主的目录、配置命令与重启方式。未适配的平台走 generic 路线：安装可移植的 CLI、MCP 或项目依赖，并明确“宿主接入方式尚未核验”。不要猜测目录、配置键或 slash 命令。

## 1. 查询并核验具体项目

按证据优先级识别：用户给出的精确 URL/包名/本地路径 → 官方仓库或包注册表 → 模糊名称搜索。模糊时只展示最相关候选（精确名、维护状态、描述、官方链接）；仍不能唯一确认才请用户选择。

核验官方来源、发布状态、许可证/维护状态、支持系统和安装形态。不要因为名称相似就安装，也不要把搜索摘要当成事实来源。

## 2. 解释用途并评估适配性

在安装前给出简洁表格：

| 项目是什么 | 常见存在形态 | 与现有体系的关系 | 对用户的实际价值 |
|---|---|---|---|
| 官方功能与边界 | skill / MCP / CLI / 库 / 二进制 / 桌面 app，或其组合 | 重名、端口、配置、依赖、权限、功能重叠与兼容性 | 能解决的具体任务、局限与替代方案 |

检查已有同名命令、MCP、skill、配置和依赖，给出“无冲突 / 可共存 / 需迁移 / 不建议安装”的结论和证据。无法确认时标记“待验证”，不能保证兼容。

## 3. 给出选择与推荐

基于上一步给出至少两个真实可行选项；若确实只有一种安装形式，另一个选项必须是“不安装/继续使用现有方案”。每个选项说明安装形态与作用域、写入目录/配置、权限和维护成本、对现有体系的影响、配置/卸载难度和适用场景。

必须有一个 **推荐选项** 并说明理由，也必须提供 **不安装** 选项。推荐不等于替用户决定：全局安装、配置写入、二进制执行、密钥/网络访问或桌面安装在执行前仍需用户选择。

## 4. 安装、验证与配置

执行用户选择的官方方案。预检版本、目标目录、系统/CPU 架构、已存在安装、认证和网络要求。用官方支持的最小真实操作验证，不以退出码为唯一依据。

能够安全、可逆且不需要用户秘密的信息配置，直接完成并报告改动；涉及授权、API Key、付费、不可逆覆盖或用户偏好的设置，给出逐步操作并停在需要用户输入的位置。

## 5. 生成说明书

默认生成 Markdown，用户需要正式交付时再导出 Word。调用 `scripts/gen_tool_manual.py` 的元数据必须包含：

- 已核验功能、常见存在形态和目标宿主；
- 兼容性评估与用户价值；
- 所有选项、唯一推荐、最终选择；
- 实际安装位置、命令与验证结果；
- 已完成/仍需用户完成的配置步骤；
- 有证据支持的调用方式；
- 官方资料与本地核验来源。

`triggers.slash`、`triggers.natural_language`、`triggers.cli`、`triggers.mcp_tools` 可为空数组。空数组表示“官方资料未声明”，绝不从项目名派生 `/项目名`、口语词或宿主指令。生成后回读文档，检查推荐方案、配置状态、调用方式提示、来源和文本。

## 宿主 adapter 约定

新增某一宿主支持前，在 `adapters/README.md` 记录并验证：识别信号、支持工具形态、作用域和写入位置、配置命令、重启/发现方法、卸载方式、系统/shell 差异及官方来源。未通过这道证据门槛前不得声称支持。

## 边界

- 不安装身份未确认的项目；不把项目名、示例或安装命令伪装成触发词。
- 不跳过冲突/价值判断或用户的方案选择。
- 不硬编码宿主、路径、操作系统、代理、镜像或工作区。
- 不把“安装成功”误报为“配置完成”或“可用”。
